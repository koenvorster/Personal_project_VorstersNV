PROBLEEM B — SEVERITY: KRITISCH
Titel: AgentRunner is een module-level singleton zonder dependency injection
Wat: In agent_runner.py:
  _runner: AgentRunner | None = None
  def get_runner() -> AgentRunner:
      global _runner
      if _runner is None:
          _runner = AgentRunner()
      return _runner
  AgentRunner.__init__() laadt bij instantiatie ALLE 32 agent YAML-files en
  initialiseert de ContextAssembler. Dit wordt aangeroepen bij eerste gebruik.
Problemen:
  1. Geen dependency injection → untestbaar zonder mock
  2. Lazy singleton is niet thread-safe bij concurrent FastAPI requests
  3. Geen hot-reload: als een YAML verandert, moet de server herstart worden
  4. AgentRunner wordt in meerdere routers gebruikt — impliciete globale state
Oplossing:
  class AgentRunner:
      def __init__(self, agents_dir: Path = AGENTS_DIR):
          self._agents: dict[str, Agent] = {}
          self._load_agents()

      @classmethod
      def from_directory(cls, path: Path) -> "AgentRunner":
          return cls(agents_dir=path)

  # In FastAPI startup:
  @asynccontextmanager
  async def lifespan(app: FastAPI):
      app.state.agent_runner = AgentRunner()
      yield

  # In routers via Depends:
  def get_agent_runner(request: Request) -> AgentRunner:
      return request.app.state.agent_runner

PROBLEEM C — SEVERITY: HOOG
Titel: code_analyse_agent heeft max_tokens: 512 — breekt de kernservice
Wat: In agents/code_analyse_agent.yml:

  Dit is de agent die LEGACY CODE ANALYSEERT — de primaire €500-€7.000 dienst.
  512 output tokens is onvoldoende voor een architectuuroverzicht, business rules
  extractie en aanbevelingen gecombineerd. Dat is ±400 woorden — een korte brief.
Impact: De duurste en meest gedifferentieerde dienst levert incompleet werk.
Oplossing:
  max_tokens: 4096  # voor volledig type
  # Of: per analyse_type configureerbaar maken
  token_config:
    architectuur: 2048
    business_rules: 3072
    kwaliteit: 2048
    volledig: 4096

PROBLEEM D — SEVERITY: HOOG
Titel: consultancy_orchestrator.yml heeft geen runtime implementatie
Wat: De orchestrator beschrijft een prachtige 3-stap pipeline (intake → parallel
  analyse → rapport) maar er is geen Python code die:
  - de "parallel: true" instructie uitvoert via asyncio.gather()
  - de input_mapping van "intake.code_analyse_vereist" vertaalt naar runtime variabelen
  - de condition "intake.code_analyse_vereist == true" evalueert
  Het is een YAML-spec zonder een YAML-runtime die hem uitvoert.
  agent_runner.py laadt enkelvoudige agents, niet orchestrator-definities.
Impact: De consultancy pipeline bestaat alleen op papier. Aanklant kan er niet mee werken.
Oplossing: Implementeer een OrchestratorRunner klasse:
  class OrchestratorRunner:
      async def run(self, orchestrator_name: str, opdracht: dict) -> OrchestratorResult:
          spec = self._load_orchestrator(orchestrator_name)
          intake = await self._run_intake(spec["intake"], opdracht)
          analyses = await self._run_parallel(spec["analyses"], opdracht, intake)
          rapport = await self._run_rapport(spec["rapport"], opdracht, intake, analyses)
          return OrchestratorResult(...)

PROBLEEM E — SEVERITY: HOOG  
Titel: Frontend root page.tsx is volledig 'use client' — SEO-lek voor portfolio
Wat: Eerste regel van frontend/app/page.tsx is 'use client'.
  Gevolg: de HOMEPAGE — het meest SEO-kritieke stuk — wordt client-side gerenderd.
  Google ziet bij crawl een lege Next.js shell totdat JS geladen is.
  De stats (4+ jaar ervaring, 15+ projecten), expertise-titels, project-namen
  worden niet geïndexeerd.
  Voor een PORTFOLIO SITE is dit een fundamentele fout.
Waarom het zo is: framer-motion + useInView + useState vereisen client context.
Oplossing:
  1. Splits animatiecomponenten in aparte Client Components
  2. Maak de pagina zelf een Server Component
  3. Gebruik Suspense boundaries voor animaties:
    // app/page.tsx — Server Component (geen 'use client')
    export default function HomePage() {
      return (
        <>
          <HeroSection /> // Server Component, statische HTML
          <Suspense fallback={<StatsGrid data={stats} />}>
            <AnimatedStatsGrid stats={stats} /> // Client Component
          </Suspense>
        </>
      )
    }

GAP 1 — KRITISCH: Geen enkel live demo van AI-agents voor bezoekers
  De /ai-lab pagina beschrijft de architectuur met mooie laagtekst maar een bezoeker
  kan NIETS uitproberen. Er is geen live chat, geen "analyseer dit code fragment" knop,
  geen "stuur een business case en krijg een rapport" demo.
  Voor een AI-consultancy portfolio is dit de grootste gemiste kans.
  Vergelijk: een web developer toont een live website. Een AI-consultant moet een
  live AI-demo tonen.

GAP 2 — KRITISCH: Geen case studies met meetbare resultaten
  DIENSTEN_AANBOD.md vermeldt het loonberekeningssysteem in 2 zinnen.
  Er is geen /case-studies route, geen before/after screenshots, geen klantcitaat,
  geen concrete resultaten ("25 looncomponenten gedocumenteerd in 3 werkdagen").
  Een klant beslist op basis van: "Heeft hij dit soort werk al gedaan? Wat leverde het op?"

GAP 3 — HOOG: Contact flow is alleen een mailto: link
  De volledige contactstrategie is: href="mailto:koen@vorsters.dev"
  Er is een /contact route maar geen contactformulier, geen Calendly-integratie,
  geen "boek een gratis 30-minuten call" knop.
  KMO-klanten sturen geen koude e-mails naar een onbekende. Ze klikken op "Plan een gesprek".

GAP 4 — HOOG: Pricing is onvoldoende zichtbaar op de diensten-pagina
  De diensten-kaarten tonen "Op maat", "€75/u", "Vanaf €500".
  De gedetailleerde prijstabellen (uit DIENSTEN_AANBOD.md) staan NIET op de website.
  KMO-klanten schrikken af van "Op maat" als ze geen idee hebben van de orde van grootte.
  Transparante indicatieprijzen verlagen de drempel voor eerste contact.

GAP 5 — HOOG: /blog route bestaat maar blog-content ontbreekt
  Een lege blog is ERGER dan geen blog: het suggereert dat het project is opgegeven.
  Oplossing: óf verwijder de route óf publiceer 3-5 artikelen:
    - "Hoe ik een Java loonberekeningssysteem analyseerde met AI in 3 dagen"
    - "Lokale AI vs cloud AI: wat kies je als KMO?"
    - "5 tekenen dat uw legacy code een documentatie-audit nodig heeft"

GAP 6 — MEDIUM: Geen social proof / testimonials
  Er zijn 0 klantcitaten, 0 LinkedIn aanbevelingen, 0 logo's van bedrijven waarvoor
  gewerkt is. De "15+ projecten afgeleverd" stat op de homepage is niet onderbouwd.
  Eén echte klantcitaat is 10x meer waard dan de mooiste techstack-badge.

GAP 7 — MEDIUM: /projecten slug-pages zijn generieke project-kaarten
  De [slug] route voor projecten bestaat maar de detailpagina's zijn template-cards
  zonder diepte. Er ontbreken: screenshots, live links, GitHub links, lessons learned,
  technische uitdagingen. Een potentiële klant wil zien HOE je denkt, niet alleen WAT je bouwt.

GAP 8 — LAAG: Geen duidelijk onderscheid B2B vs B2C
  De shop/webshop-routes suggereren B2C maar alle diensten zijn B2B.
  Dit creëert ruis bij een professionele bezoeker die zoekt naar een IT-partner.

6.1 AUTHENTICATIE — Meest urgente blokkeer voor klantportaal
────────────────────────────────────────────────────────────────────────────────

AANBEVELING: Vervang Keycloak (dev-only) door NextAuth.js voor snelle productie

Implementatie:
  1. npm install next-auth @auth/prisma-adapter
  2. Configureer providers: Email (magic link) + LinkedIn OAuth
  3. Bescherm /portal/* routes via middleware.ts:
     export { default } from "next-auth/middleware"
     export const config = { matcher: ["/portal/:path*"] }
  4. Backend: verify JWT token in FastAPI auth.py
     (NextAuth JWT is compatibel met FastAPI python-jose)

Reden voor NextAuth over Keycloak:
  - Keycloak vereist een draaiende server (Docker of hosted)
  - NextAuth is embedded in de Next.js app — geen extra infrastructuur
  - Voor <100 klanten is Keycloak overkill
  - Keycloak later altijd nog als enterprise-upgrade
