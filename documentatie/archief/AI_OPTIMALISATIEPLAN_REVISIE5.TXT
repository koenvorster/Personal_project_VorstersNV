 SECTIE 2: KRITISCHE GAP ANALYSE — VOORBIJ REVISIE 4

Na een diepgaande codebase-inspectie zijn de volgende ECHTE gaps gevonden
die NIET in de bestaande G-01..G-31 catalogus staan:

──────────────────────────────────────────────────────────────────────────────
 KLASSE A: ARCHITECTURELE LACUNES (blokkerend voor productie)
──────────────────────────────────────────────────────────────────────────────

G-32 | GEEN CLIENT PROJECT ISOLATION
  Beschrijving: Alle agents, logs en context zijn single-tenant. Twee
    klantprojecten die parallel lopen kunnen elkaars context lekken.
    consultancy_orchestrator.yml logt naar logs/{klant_naam}_{datum}.md
    maar heeft GEEN namespace-isolatie, GEEN toegangscontrole,
    GEEN resource quota per klant.
  Impact: KRITIEK — datahygiëne, GDPR-risico, concurrentieel risico
  Gap severity: BLOCKER voor meerdere simultane klanten

G-33 | GEEN VECTOR DATABASE / SEMANTISCHE RETRIEVAL
  Beschrijving: memory.py doet alleen turn-based summarization maar slaat
    GEEN embeddings op. Er is geen vector store (pgvector, Chroma, Qdrant).
    Code-analyse resultaten van vorige projecten zijn niet terugvindbaar.
    klant_rapport_agent genereert telkens vanuit nul — geen accumulatie.
  Impact: HOOG — elke analyse begint opnieuw, kennis verdwijnt
  Gap severity: BLOCKER voor kennisaccumulatie

G-34 | GEEN ADAPTIVE CONTEXT WINDOW MANAGEMENT
  Beschrijving: java_chunk_analyse_agent.yml heeft chunking-logica voor
    Java bestanden, maar dit is hardcoded per agent. Er is geen generieke
    AdaptiveChunker die:
      - automatisch chunks bepaalt op basis van model context window
      - prioriteert welke chunks het meest relevant zijn
      - overlap beheert tussen chunks (sliding window)
      - context compresseert bij overschrijding
    Op een CPU-only laptop met llama3 (4096 tokens) is dit kritiek.
  Impact: HOOG — grote codebases falen of leveren slechte resultaten
  Gap severity: BLOCKER voor enterprise code-analyse

G-35 | GEEN AGENTIC RAG PIPELINE
  Beschrijving: Geen Retrieval-Augmented Generation. Wanneer een agent
    kennis nodig heeft over een klant of codebase, heeft hij ENKEL wat
    in de prompt zit. Geen retrieval van:
      - eerder geanalyseerde klantcode
      - sector-specifieke benchmarks
      - gelijkaardige processen uit vorige opdrachten
      - VorstersNV kennisbank
  Impact: HOOG — kwaliteit van rapporten schaalt niet met ervaring
  Gap severity: BLOCKER voor differentiatie als consultancyplatform

G-36 | GEEN AGENT VERSIONING SYSTEEM
  Beschrijving: 26 YAML agent-bestanden hebben version: "1.0" maar er is
    GEEN systeem voor:
      - semantic versioning van prompts (major.minor.patch)
      - diff tussen promptversies
      - rollback naar vorige versie bij regressie
      - A/B testing van promptversies op productietrafiek
      - audit trail: welke promptversie gaf welk resultaat
    ab_tester.py bestaat maar is niet gekoppeld aan het agent versioning.
  Impact: HOOG — onveilig prompts updaten in productie
  Gap severity: HOOG

G-37 | GEEN COST FORECASTING VOOR OFFERTES
  Beschrijving: cost_governance.py meet verbruik maar voorspelt NIET.
    Bij een nieuwe klantaanvraag kan VorstersNV NIET automatisch zeggen:
    "Deze code-analyse zal ~€X kosten en ~Y uur duren."
    Er is geen:
      - tokenschatting pre-run op basis van inputgrootte
      - historisch verbruik per opdrachtstype
      - confidence interval op de schatting
      - offertemodule die AI-kosten vertaalt naar klantprijs
  Impact: HOOG — handmatige offertes, onderschatting van AI-kosten
  Gap severity: HOOG

 KLASSE B: COMPLIANCE & GOVERNANCE LACUNES

G-38 | GEEN EU AI ACT RISICOKLASSIFICATIE
  Beschrijving: EU AI Act trad volledig in werking in 2026. VorstersNV
    levert AI-systemen aan Belgische KMOs. Dit vereist:
      - risicoklassificatie van elke capability (limited/high/unacceptable)
      - conformiteitsdocumentatie voor high-risk AI systemen
      - transparantieverplichtingen (klant weet wanneer AI beslist)
      - menselijk toezicht bij high-risk beslissingen
    Huidige HITL-policies zijn onvoldoende voor EU AI Act compliance.
  Impact: JURIDISCH RISICO — boetes tot €30M of 6% omzet
  Gap severity: KRITIEK (tijdsgevoelig)

G-39 | GEEN GEAUTOMATISEERDE PII DETECTIE / MASKING
  Beschrijving: Klanten leveren vaak productiedumps aan met echte
    klantgegevens (namen, BSN, IBAN, emailadressen). Er is geen:
      - automatische PII-scanner vóór code-analyse
      - masking engine (John Doe → KLANT_001)
      - GDPR data-minimisation check op agent outputs
      - data lineage: waar in het rapport komt PII vandaan?
    GDPR artikel 5(1)(c) vereist data-minimalisatie.
  Impact: KRITIEK — GDPR-boete risico bij klantprojecten
  Gap severity: BLOCKER

G-40 | GEEN NIS2 SECURITYCONTROLES
  Beschrijving: VorstersNV is een IT-dienstverlener en valt dus onder
    NIS2 als "important entity". Vereiste controles ontbreken:
      - incident response procedure voor AI-gerelateerde incidenten
      - supply chain security assessment (Ollama, externe APIs)
      - vulnerability disclosure policy
      - security audit logging conform NIS2
      - business continuity plan voor AI-platform
  Impact: HOOG — NIS2-non-compliance door CERT.be
  Gap severity: HOOG

G-41 | GEEN BELGISCHE SECTOR-SPECIFIEKE KENNISBANK
  Beschrijving: Voor de Belgische KMO-markt ontbreekt domeinkennis over:
      - Belgische boekhoudregels (Boekhoudwet 2019)
      - RSZ-bijdragen en SWT-regels (loonberekening)
      - Sectoriële CAOs (PC 200, PC 218, PC 307 etc.)
      - NBB-rapporteringsverplichtingen
      - Kruispuntbank voor Ondernemingen (KBO) patterns
    agents/bedrijfsproces_agent.yml bevat sector: [hr_payroll, boekhouding]
    maar heeft GEEN feitelijke regelgeving ingebakken.
  Impact: MEDIUM — verkeerde adviezen voor Belgische context
  Gap severity: MEDIUM (maar competitief kritiek)

 KLASSE C: AI/LLM TECHNOLOGIE LACUNES (2025/2026 best practices)

G-42 | GEEN MIXTURE-OF-AGENTS VOOR COMPLEXE OPDRACHTEN
  Beschrijving: Voor complexe consultancy-opdrachten wordt één agent
    gebruikt (bijv. code_analyse_agent). Moderne MoA-patronen gebruiken
    meerdere "proposer" agents die parallel een antwoord genereren,
    gevolgd door een "aggregator" die het beste antwoord synthetiseert.
    Dit verhoogt kwaliteit significant voor moeilijke vragen.
  Impact: HOOG — suboptimale rapporten voor complexe architectuurvragen
  Gap severity: HOOG (lage implementatiedrempel)

G-43 | GEEN STRUCTURED REASONING (CHAIN-OF-THOUGHT) LOGGING
  Beschrijving: Agents geven output maar hun redeneerpad is niet
    zichtbaar of opgeslagen. Voor consultancy-rapporten is het
    redeneerpad ZELF waardevol (klant begrijpt WAAROM een aanbeveling).
    Reasoning tokens (DeepSeek R1, Qwen3-thinking stijl) worden
    niet apart opgeslagen of geëxposeerd.
  Impact: MEDIUM — black-box rapport, minder vertrouwen bij klant
  Gap severity: MEDIUM

G-44 | GEEN MULTIMODALE AGENT CAPABILITIES
  Beschrijving: Consultancyrapporten zijn beter met visuele elementen:
      - architectuurdiagrammen (Mermaid/PlantUML uit agent output)
      - swimlane diagrammen voor procesanalyse
      - data-flow diagrammen uit code-analyse
    bedrijfsproces_agent.yml heeft swimlane_diagram_genereren als
    capability maar er is geen renderer/exporter.
    code_analyse_agent.yml genereert diagram_tekst (string) maar
    dit wordt nooit omgezet naar een echt diagram.
  Impact: MEDIUM — tekstuele rapporten minder professioneel
  Gap severity: MEDIUM

G-45 | GEEN SELF-IMPROVEMENT LOOP OP BASIS VAN KLANTFEEDBACK
  Beschrijving: evaluation/metrics (volledigheid_score, etc.) zijn
    gedefinieerd in agent YAMLs maar er is geen mechanisme om:
      - klantfeedback te koppelen aan specifieke promptversies
      - automatisch zwakke agents te identificeren
      - prompt-verbeteringen voor te stellen op basis van patterns
      - A/B testresultaten te vertalen naar definitieve promotie
    ab_tester.py en maturity_engine.py bestaan maar zijn niet
    verbonden via een learning feedback loop.
  Impact: HOOG — platform leert niet, kwaliteitsplateau
  Gap severity: HOOG

G-46 | GEEN ASYNC STREAMING VOOR LANGE AGENT RUNS
  Beschrijving: agent_runner.py roept Ollama synchroon aan. Bij een
    grote code-analyse (30+ minuten) heeft de gebruiker geen
    voortgangsindicatie. Geen:
      - server-sent events (SSE) van tussentijdse agent output
      - partial result streaming naar frontend
      - progress percentages per workflow stap
      - cancellable long-running tasks
  Impact: HOOG — slechte UX, gebruiker weet niet of systeem werkt
  Gap severity: HOOG

──────────────────────────────────────────────────────────────────────────────
 4.1  CLIENT PROJECT SPACE
──────────────────────────────────────────────────────────────────────────────

MODULE:  ollama/client_project_space.py

  @dataclass
  class ClientProject:
      project_id:    str  # UUID
      client_id:     str  # Keycloak user/org ID
      project_name:  str
      sector:        str
      data_class:    DataClassification  # PUBLIC | INTERNAL | CONFIDENTIAL
      resource_quota: ProjectQuota
      created_at:    datetime
      status:        ProjectStatus  # ACTIVE | PAUSED | COMPLETED | ARCHIVED

  @dataclass
  class ProjectQuota:
      max_tokens_per_run:    int = 100_000
      max_runs_per_day:      int = 50
      max_storage_mb:        int = 500
      max_concurrent_agents: int = 3

ISOLATIEMECHANISMEN:
  1. Database: Row-Level Security (RLS) in PostgreSQL
     → elke query gefilterd op project_id
     → agent kan NOOIT data van ander project zien
  2. Logs: /logs/{project_id}/ (geen gedeelde log map)
  3. VectorStore: namespace per project_id in pgvector
  4. Memory: sessie-scoped aan project_id
  5. Decision Journal: project_id als verplicht veld

PII SCANNER (pre-processing):
  Engine: presidio (Microsoft, open-source)
  Entiteiten: EMAIL, PHONE_NUMBER, IBAN, BSN/RRN, IP_ADDRESS,
              PERSON, LOCATION, ORGANIZATION
  Actie bij detectie:
    CONFIDENTIAL project → BLOKKEER of MASKEER (configureerbaar)
    INTERNAL project    → MASKEER automatisch, log in audit trail
    PUBLIC project      → LOG waarschuwing, ga door

  pii_result = pii_scanner.scan(input_text)
  clean_input = pii_scanner.mask(input_text, pii_result)
  # Origineel NOOIT doorgegeven aan LLM

──────────────────────────────────────────────────────────────────────────────
 4.2  AGENTIC RAG ENGINE
──────────────────────────────────────────────────────────────────────────────

MODULE:  ollama/rag_engine.py

ARCHITECTUUR:
  ┌─────────────────────────────────────────────────────────────────┐
  │  RAG PIPELINE                                                   │
  │                                                                 │
  │  INGEST FASE (na elke succesvolle analyse):                     │
  │    1. Extract findings (business_rules, aanbevelingen)          │
  │    2. Genereer embeddings via sentence-transformers              │
  │       (model: all-MiniLM-L6-v2, lokaal draaiend, CPU-compatibel)│
  │    3. Sla op in pgvector met metadata:                          │
  │       {project_id, sector, agent_name, finding_type, timestamp} │
  │    4. Update KnowledgeGraph nodes/relaties                      │
  │                                                                 │
  │  RETRIEVAL FASE (vóór elke agent call):                         │
  │    1. Genereer query-embedding van huidige taak                 │
  │    2. Cosine similarity search in pgvector (top-K=5)            │
  │    3. Re-rank resultaten (cross-encoder of MMR diversity)       │
  │    4. Filter op project_id + data_classification                │
  │    5. Format als [RETRIEVED_CONTEXT] sectie in prompt           │
  └─────────────────────────────────────────────────────────────────┘

VECTOR STORE SCHEMA (pgvector):
  CREATE TABLE rag_documents (
    id          UUID PRIMARY KEY,
    project_id  UUID NOT NULL,          -- Row-Level Security
    client_id   UUID NOT NULL,
    sector      TEXT,
    agent_name  TEXT,
    doc_type    TEXT,                   -- code_pattern|business_rule|benchmark
    content     TEXT,
    embedding   vector(384),            -- all-MiniLM-L6-v2 dimensie
    metadata    JSONB,
    created_at  TIMESTAMPTZ DEFAULT NOW()
  );
  CREATE INDEX ON rag_documents USING ivfflat (embedding vector_cosine_ops);

RETRIEVAL TYPES:
  1. SEMANTIC: zoek op betekenis (voor open vragen)
  2. HYBRID:   semantic + keyword BM25 (voor technische termen)
  3. FILTERED: semantic binnen specifiek sector of agent_name
  4. GRAPH:    graph-hop via KnowledgeGraph voor gerelateerde entiteiten

CROSS-PROJECT RETRIEVAL (opt-in per client):
  - Klant kan toestaan dat anonieme findings gedeeld worden
  - Sector-benchmarks worden opgebouwd uit geanonimiseerde projectdata
  - "VorstersNV heeft X soortgelijke projecten geanalyseerd" → kwaliteit ↑

──────────────────────────────────────────────────────────────────────────────
 4.3  ADAPTIVE CHUNKER
──────────────────────────────────────────────────────────────────────────────

MODULE:  ollama/adaptive_chunker.py

PROBLEEM:
  CPU-only llama3 = max ~4096 tokens context window
  llama3.1:8b = max 128k (maar memory-intensief op CPU)
  Enterprise codebase = 500k+ tokens

STRATEGIE:
  ┌─────────────────────────────────────────────────────────────────┐
  │  INPUT: bestand of code_fragment                                 │
  │     ↓                                                           │
  │  STAP 1: Token estimatie (tiktoken, no LLM needed)              │
  │     → bepaal chunk_size = model_context * 0.6 (safety margin)   │
  │     ↓                                                           │
  │  STAP 2: Semantische grenzen detecteren                         │
  │     → splits op klasse/functie/method grenzen (AST-aware)       │
  │     → NOOIT splits in het midden van een functie                │
  │     ↓                                                           │
  │  STAP 3: Relevantie scoring per chunk                           │
  │     → TF-IDF op basis van analyse_type (business_rules/arch)    │
  │     → Top-N chunks voor beschikbaar context budget              │
  │     ↓                                                           │
  │  STAP 4: Overlap management                                     │
  │     → 10% overlap tussen opeenvolgende chunks                   │
  │     → ChunkManifest: houdt bij welke chunks verwerkt zijn       │
  │     ↓                                                           │
  │  STAP 5: Progressieve synthese                                  │
  │     → per chunk: partial_findings[]                             │
  │     → na alle chunks: synthesize_agent samenvoegen              │
  └─────────────────────────────────────────────────────────────────┘

  @dataclass
  class ChunkManifest:
      total_chunks:    int
      processed:       list[int]
      partial_findings: list[dict]
      synthesis_ready: bool

LANGUAGE-AWARE SPLITTING:
  - Java:    splits op class/interface/method niveau (via javalang)
  - Python:  splits via ast.parse (class/def grenzen)
  - C#:      splits via regex op namespace/class/method
  - Generic: splits op blanco regels + indentation niveau

DEPLOYMENT LIFECYCLE:
  DRAFT → SHADOW → CANARY (5%) → STABLE → DEPRECATED → ARCHIVED

ROLLBACK MECHANISME:
  1. Detecteer regressie: eval_score daalt > 10% t.o.v. vorige versie
  2. Auto-rollback trigger: maturity_engine.py detecteert regressie
  3. Control plane schakelt terug naar laatste STABLE versie
  4. Incident report in decision journal

SHADOW MODE (veiligste manier om te testen):
  - Nieuwe versie ontvangt zelfde input als productie-versie
  - Output wordt NIET gebruikt, enkel vergeleken
  - Score-vergelijking na 100 runs → promotie of reject
  - Geen enkel productie-risico

──────────────────────────────────────────────────────────────────────────────
 4.8  MIXTURE-OF-AGENTS (MoA) ORCHESTRATOR
──────────────────────────────────────────────────────────────────────────────

MODULE:  ollama/mixture_of_agents.py

WANNEER ACTIVEREN:
  - analyse_diepte == "diepgaand"
  - OR capability.risk_level == "HIGH"
  - OR confidence_required > 0.90

PATROON:
  ┌─────────────────────────────────────────────────────────────────┐
  │  PROPOSER FASE (parallel, temperature=0.4)                      │
  │    Agent A: code_analyse_agent (perspective: architectuur)      │
  │    Agent B: security_permissions_agent (perspective: security)  │
  │    Agent C: clean_code_reviewer_agent (perspective: kwaliteit)  │
  │                    ↓ (alle 3 parallel)                          │
  │  AGGREGATOR FASE (temperature=0.2)                              │
  │    klant_rapport_agent:                                         │
  │      - Ontvang alle 3 perspectieven                             │
  │      - Identificeer consensus vs conflicterende meningen        │
  │      - Synthetiseer beste antwoord                              │
  │      - Vermeld dissenting views in rapport                      │
  └─────────────────────────────────────────────────────────────────┘

VOORDELEN:
  - Hogere kwaliteit door meerdere perspectieven
  - Conflicten worden zichtbaar (transparantie)
  - Niet afhankelijk van één agent's blinde vlekken
  - Parallel = sneller dan sequentieel

COST IMPACT:
  3x tokens verbruik voor proposers + 1x aggregator ≈ 4x kostprijs
  Enkel activeren bij diepgaande analyses (gerechtvaardigd)

 8.2  OLLAMA / LOCAL LLM OPTIMALISATIE
──────────────────────────────────────────────────────────────────────────────

HUIDIG (CPU-only laptop):
  - llama3:8b          ← code analyse (goed voor redeneren)
  - mistral:7b         ← rapport generatie (sterk in taal)
  - codellama:7b       ← specifiek code begrip
  - all-MiniLM-L6-v2  ← embeddings (via sentence-transformers, geen Ollama)

CPU OPTIMALISATIE TIPS:
  1. Gebruik Ollama num_ctx parameter bewust:
       llama3 standaard: 2048 → voor code analyse: 4096
       (hogere ctx = meer RAM gebruik op CPU)
  2. llm_load_time: pre-load meest gebruikte modellen bij startup
       ollama.pull("llama3") bij applicatiestart
  3. Parallel requests: max 1 op CPU (num_parallel=1 in Ollama config)
  4. Quantisatie: gebruik Q4_K_M (beste kwaliteit/snelheid balans op CPU)
       ollama pull llama3:8b-instruct-q4_K_M

OVERGANG NAAR GPU (gaming desktop):
  Aanbevolen workflow bij GPU beschikbaar:
  1. OLLAMA_HOST=0.0.0.0 op gaming desktop (server mode)
  2. Laptop stuurt requests naar gaming desktop via lokaal netwerk
  3. OLLAMA_BASE_URL=http://192.168.x.x:11434 in .env
  4. Modellen met GPU:
       llama3.1:70b (Q4)  ← voor MoA aggregator (70B >> 8B kwaliteit)
       mixtral:8x7b        ← voor complexe code analyse
       codellama:34b       ← voor enterprise Java/C# analyse
  5. Platform_adapter.py: route per capability naar juiste host
       HIGH-RISK capability → GPU desktop
       MINIMAL-RISK query  → CPU laptop (sneller voor kleine taken)

CONTEXT WINDOW AANPAK:
  Model               Max ctx     Aanbevolen gebruik
  ──────────────────────────────────────────────────
  llama3:8b           4096        Kleine code, rapporten < 2000 tokens
  llama3.1:8b         131072      Grote code-analyses op CPU (traag!)
  llama3.1:70b (GPU)  131072      Enterprise analyses (ideaal)
  codellama:34b (GPU) 16384       Gespecialiseerde code-analyse
  mistral:7b          32768       Rapport generatie, lange outputs

     → Nodig voor: klant_rapport_agent output

 SECTIE 14: TECHNISCHE BESLISSINGEN REVISIE 5
================================================================================

VECTOR STORAGE:
  Keuze: pgvector (PostgreSQL extensie)
  Reden: geen extra infra, bestaande PostgreSQL 16, RLS-isolatie werkt
  Fallback: Chroma (in-memory voor lokale dev/test zonder PostgreSQL)
  Embedding model: all-MiniLM-L6-v2 (384 dimensies, CPU-snel, 90MB)

PII SCANNING:
  Keuze: Microsoft Presidio (open-source, lokaal, geen API calls)
  Reden: GDPR-compliant (data verlaat NOOIT het systeem), gratis
  Belgische entiteiten: RRN-nummers, IBAN (BE...), BTW-nummers (BE...)

KNOWLEDGE GRAPH (eerste versie):
  Keuze: PostgreSQL JSONB + recursive CTE queries
  Reden: geen extra infra, bewezen schaalbaar tot 1M nodes
  Upgrade pad: Neo4j bij > 5M nodes of als graph-queries te complex worden

STREAMING:
  Keuze: Server-Sent Events (SSE) via FastAPI EventSourceResponse
  Reden: eenvoudiger dan WebSockets, HTTP/2 compatible, browser-native
  Frontend: EventSource API in Next.js (geen extra library)

AGENT VERSIONING:
  Keuze: directory-based (agents/versions/{name}/{version}/)
  Reden: git-compatible, eenvoudig, geen extra DB tabel nodig
  Semver: bump via CLI script (scripts/bump_agent_version.py)

COMPLIANCE FRAMEWORK:
  EU AI Act: zelfgebouwde lichte engine (geen extern product)
    Reden: vereisten zijn relatief eenvoudig voor bestaande capabilities
    Scope: risicoklassificatie + HITL + audit logging
  GDPR: Presidio + bestaande decision_journal.py retention policy
  NIS2: structlog JSON logging + incident response procedure (YAML)

BACKWARDS COMPATIBILITY:
  Alle Revisie 4 modules: ONGEWIJZIGD (uitbreiden, niet vervangen)
  Agent YAMLs: nieuwe secties zijn OPTIONEEL (backward compatible)
  Database: additive migraties (ALTER TABLE ADD COLUMN, nooit DROP)
  API: nieuwe endpoints, bestaande /api/... ongewijzigd
