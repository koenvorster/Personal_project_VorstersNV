 SECTIE 3: CONTROL PLANE — KERN VAN HET PLATFORM
================================================================================

OLLAMA/CONTROL_PLANE.PY — ROUTING CRITERIA
────────────────────────────────────────────
  Control plane beslist NIET enkel op event.type
  Maar op 8 dimensies tegelijk:

  Dimensie           Bron
  ─────────────────────────────────────────────
  event.type         EventType enum (orders.py)
  environment        dev | test | prod
  risk_level         0-100 (fraude score, order waarde)
  cost_budget        capability YAML: cost_budget_monthly_eur
  latency_budget     SLA tier (bronze/silver/gold)
  capability_maturity L1-L4
  model_availability ollama health check
  policy_rules       /policies/*.yml

ROUTING LOGICA
───────────────
  select_capability(event)
    → zoekt in capability registry op event.type + context
    → controleert maturity level (L1 = enkel in dev)
    → controleert feature flag (rollout ring)

  select_execution_path(capability, context)
    → kiest workflow lane (DETERMINISTIC/ADVISORY/GENERATIVE/ACTION)
    → kiest agent(s) uit fleet
    → kiest model (cheap → premium bij escalatie)

  enforce_policy(capability, context)
    → laadt /policies/*.yml
    → blokkeert bij policy violation
    → logt violation naar audit trail

  check_budget(capability, usage)
    → vergelijkt maandelijks verbruik met cost_budget_monthly_eur
    → bij overschrijding: fallback naar cheap model of HITL

  determine_rollout_ring(context)
    → Ring 0: lokaal, Ring 1: AI team, Ring 2: intern, Ring 3: beperkt, Ring 4: full

  requires_human_approval(capability, risk)
    → HITL-001: environment=prod + bepaalde capabilities
    → HITL-002: risk_score >= 75
    → HITL-003: ring < 2 + business-critical capability

 SECTIE 4: CAPABILITY MATURITY MODEL
================================================================================

Elke capability YAML krijgt 4 nieuwe secties:

  maturity:
    level: L2                        # L1=experimental, L2=internal-beta,
    levels:                          # L3=team-production, L4=business-critical
      L1: experimental
      L2: internal-beta
      L3: team-production
      L4: business-critical

  assurance:
    eval_required: true
    human_approval_required: conditional   # always | conditional | never
    min_schema_validation_rate: 0.95
    min_first_pass_score: 0.85

  operational:
    owner: team-ai-platform
    sla_tier: bronze                 # bronze | silver | gold
    cost_budget_monthly_eur: 150

  release:
    rollout_ring: ring-1             # ring-0..ring-4
    feature_flag: ai.capability.order-validation.v4

MATURITY GATES (capability gaat pas naar volgend level als):
  L1 → L2: unit evals > 80%, schema validation rate > 90%
  L2 → L3: capability evals > 85%, 1 week stabiel in AI team
  L3 → L4: chain + business evals OK, rollback getest, observability actief
  L4:       continu gemonitord, SLA gegarandeerd, alerting actief

 SECTIE 6: WORKFLOW LANES
================================================================================

4 LANES — elke step declareert zijn lane:

  A. DETERMINISTIC LANE
     Doel:     schema validatie, contract mapping, data normalisatie
     Kenmerken: temperature=0.0, strict schema, geen creativiteit, snel
     Agents:   schema_validator, contract_mapper, data_normalizer
     QG:       schema compliance = 100% (hard gate)

  B. ADVISORY LANE
     Doel:     impact analyse, architectuur advies, fraude detectie
     Kenmerken: explainability verplicht, confidence score (0-100), reviewbaar
     Agents:   fraud-advisor, order-analyst, risk-classifier, architect_agent
     QG:       confidence score >= 60, rationale aanwezig

  C. GENERATIVE LANE
     Doel:     content generatie, e-mails, klantenservice reacties
     Kenmerken: stijlregels actief, SEO checks, review loop (max 2 iteraties)
     Agents:   product-content-agent, email_template, klantenservice-advisor
     QG:       QG-CONTENT-01 (SEO), QG-STYLE-01 (tone of voice)

  D. ACTION LANE
     Doel:     tool calls uitvoeren, order blokkeren, account aanmaken
     Kenmerken: HITL verplicht (indien policy), audit logging, idempotent
     Agents:   order_verwerking, retour_verwerking, fraude_actie
     QG:       idempotency check, HITL cleared, audit logged

 SECTIE 7: CONTEXT ENGINEERING MODEL
================================================================================

NIEUWE CONTEXT STRUCTUUR (geordend):
──────────────────────────────────────
  [SYSTEM_POLICY]          ← actieve policy regels voor deze capability
  [PROJECT_CONTEXT vX.Y]   ← vorstersnv-context.yaml (versioned, timestamped)
  [DOMAIN_RULES]           ← BTW, herroepingstermijn, Mollie statussen
  [CAPABILITY_CONTRACT]    ← InputContract voor deze specifieke aanroep
  [TASK_INPUT]             ← de eigenlijke taak (schoon, gefilterd)
  [TOOL_RESULTS]           ← via schema + filter (NOOIT raw tool output)
  [MEMORY_SUMMARY]         ← auto-summary bij sessies > 10 turns
  [OUTPUT_CONTRACT]        ← verwacht output formaat (Pydantic schema)

REGELS:
  - Tool output nooit raw injecteren → altijd via schema + filter
  - Context altijd versioned: timestamp + version in header
  - Delimiter isolatie: [MEMORY] [TOOL_RESULT] [DOMAIN_RULES]
  - Context decay: stale context (> 8 uur) wordt automatisch vernieuwd
  - Hierarchical: main context + task-specific snippets (niet alles altijd)

3 CONTRACT TYPES:

      trace_id:            str
      timestamp:           datetime
      agent_name:          str
      model_used:          str

ALLE CONTRACTS: to_prompt_input() + from_agent_output() + validate()
Implementatie: Pydantic v2 (contract-first pattern 2025)

 SECTIE 9: EVALUATION HIERARCHY (4 LEVELS)
================================================================================

  LEVEL 1 — UNIT EVAL (per agent aanroep)
    Vraag:  Is de output valid?
    Checks: schema correct, geen empty output, geen hallucination signalen
    Trigger: elke aanroep (automatisch via schema_validator.py)
    Tool:   ollama/schema_validator.py

  LEVEL 2 — CAPABILITY EVAL (per capability run)
    Vraag:  Is de juiste output gegeven? Juiste tool gekozen?
    Checks: output matches expected (evals dataset), tool calls correct
    Trigger: bij elke prompt-versiewijziging (CI)
    Tool:   ollama/evals/judge.py (LLM-as-judge)

  LEVEL 3 — CHAIN EVAL (per workflow run)
    Vraag:  Is de juiste flow gevolgd? Werkt checkpoint recovery?
    Checks: state transitions correct, HITL triggered correct, no silent failures
    Trigger: bij elke chain configuratie wijziging
    Tool:   ollama/evals/hierarchy.py (chain evaluator)

  LEVEL 4 — BUSINESS EVAL (productie monitoring)
    Vraag:  Zijn de beslissingen correct? Hoeveel false positives?
    Checks: fraud_accuracy, human_escalation_rate, cost_per_outcome, customer impact
    Trigger: dagelijks rapport + alert bij regressie
    Tool:   ollama/quality_monitor.py + Grafana N3 dashboard

RESULTAAT: niet enkel score > 0.8, maar het systeem werkt correct end-to-end

 SECTIE 10: DEPLOYMENT RINGS
================================================================================

  Ring 0 → lokaal (developer machine, geen data)
  Ring 1 → AI team (team review, test data)
  Ring 2 → interne users (medewerkers, staging data)
  Ring 3 → beperkte productie (5-10% traffic, real data)
  Ring 4 → full productie (100% traffic)

CAPABILITY GAAT PAS DOOR NAAR VOLGEND RING ALS:
  ✓ Eval suite van huidig level geslaagd
  ✓ Policy violations = 0 in vorige ring
  ✓ Observability actief (alle 3 lagen)
  ✓ Rollback succesvol getest
  ✓ Owner heeft akkoord gegeven

FEATURE FLAGS: capability YAML bevat release.feature_flag
  Control plane controleert ring voor elke capability aanroep

 SECTIE 11: DECISION JOURNAL
================================================================================

OPSLAG: PostgreSQL tabel decision_journal + ollama/decision_journal.py

VELDEN:
  trace_id          ← koppeling met OpenTelemetry
  capability        ← welke capability werd gebruikt
  agent             ← welke agent deed het werk
  model             ← welk model (haiku/sonnet/opus)
  tools             ← welke tools werden aangeroepen
  selection_reason  ← waarom deze agent/model gekozen (keuzereden)
  alternatives      ← welke alternatieven overwogen maar niet gekozen
  human_override    ← was er menselijke interventie?
  verdict           ← APPROVED | CHANGES_REQUESTED | NEEDS_DISCUSSION
  timestamp

DOEL:
  - Explainability: waarom deze beslissing?
  - Debugging: wat ging er mis?
  - Audits: compliance/GDPR bewijs
  - Retrospectives: patroonanalyse voor verbetering

 SECTIE 12: COST & MODEL GOVERNANCE
================================================================================

PER CAPABILITY YAML:
  cost:
    max_input_tokens:   12000
    max_output_tokens:  1500
    preferred_model:    haiku       # altijd eerst proberen
    escalation_model:   sonnet      # bij retry, lage confidence, hoog risico
    max_tool_calls:     3

LOGICA:
  1. Probeer altijd preferred_model (goedkoop)
  2. Escaleer naar escalation_model bij:
     - confidence < 0.6
     - retry_count > 1
     - risk_level > 70
     - schema validation failed
  3. Fallback bij timeout (max_latency_ms overschreden)
  4. Blokkeer bij budget overschrijding (COST-002 policy)

RAPPORT: cost per capability per dag → Grafana N3 dashboard

 SECTIE 13: OBSERVABILITY — 3 LAGEN MODEL
================================================================================

  N1 — REQUEST (per LLM aanroep)
     tokens: prompt_tokens, completion_tokens
     latency: time_to_first_token, total_ms
     model: gen_ai.request.model, gen_ai.response.finish_reason
     OTel: gen_ai.* semantic conventions

  N2 — AGENT (per agent run)
     agent.name, agent.group (G1..G6)
     capability: welke capability
     fallback: was er een fallback model?
     verdict: APPROVED | CHANGES_REQUESTED
     selection_reason: waarom deze agent?
     OTel: agent.* custom attributes

  N3 — BUSINESS (dagelijks rapport)
     fraud_accuracy: % correcte fraude detecties
     onboarding_success_rate: % orders correct verwerkt
     human_escalation_rate: % HITL triggers
     cost_per_outcome: EUR per succesvolle transactie
     false_positive_rate: % onterecht geblokeerde orders
     OTel → Grafana dashboard: business KPIs

EXPORT: configureerbaar (Jaeger | Grafana Cloud | Azure Monitor)
