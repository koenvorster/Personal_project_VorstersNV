1. ARCHITECTUUROVERZICHT
================================================================================

De LPB Rekenmotor is opgebouwd als een CHAIN OF RESPONSIBILITY patroon.
Elke berekeningsstap is een zelfstandig RekenCommand-object dat via een ketting
aan elkaar gelinkt is. Een centrale RekenContext (een typed key-value store)
wordt doorheen de hele ketting doorgegeven.

Hoofdcomponenten:
  ┌─────────────────────────────────────────────────────────┐
  │  StandaardRekenmotorService / StandaardRekenmotorWebService │
  │  (entry points — Spring-beans)                          │
  └──────────────────────┬──────────────────────────────────┘
                         │ bouwt ketting via
  ┌──────────────────────▼──────────────────────────────────┐
  │  RekenChainFactoryImp                                   │
  │  implementeert RekenChainFactory                        │
  │  → maakt alle RekenCommand-instanties + linkt ze        │
  └──────────────────────┬──────────────────────────────────┘
                         │ executes
  ┌──────────────────────▼──────────────────────────────────┐
  │  RekenContext  (typed key-value store)                  │
  │  Keys: RekenBrutoKey / RekenBvKey / RekenRszKey /       │
  │        RekenBbKey / RekenNettoKey / ContextKey ...      │
  └─────────────────────────────────────────────────────────┘

RekenContext-hiërarchie:
  mainContext (persoonsniveau)
  └── opdrachtContext (per Opdracht / tewerkstelling)
      └── LoonElement (per looncode)

Basispatroon van elke RekenCommand:
  - execute(RekenContext ctx) → retourneert CONTINUE_PROCESSING of STOP
  - AbstractRekenCommand biedt gemeenschappelijke hulpmethoden
  - AbstractBerekenBv biedt de BV-berekeningsbasis

4.2  FASE BRUTO
──────────────────────────────────────────────────

BerekenBrutoLoon
  Input:   RekenContext met Persoon.getOpdrachten()
  Output:  BRUTO_MAANDLOON_KEY, BASIS_SOCIALE_INHOUDINGEN_KEY (per opdracht)
  Aanroept:
    berekenVergoedingen.bereken(context, opdracht)  → BerekenVergoedingen
    berekenSupplementen.bereken(context, opdracht)  → BerekenSupplementen
    flexiJobs.doeControles(...)
    flexiJobs.vormVakantieGeld(...)
  Formule:
    brutoMaandLoon = Σ loonElementen.getPropertyMoney(BRUTO_MAANDLOON_KEY)
    afgerond op 2 decimalen

BerekenVergoedingen
  Verwerkt alle loon-elementen van type vergoeding per opdracht
  Zet BRUTO_MAANDLOON_KEY per LoonElement

BerekenSupplementen
  Verwerkt supplementen bovenop het basisloon

BerekenVergoedingFeestDag
  Berekent feestdagvergoeding als apart loon-element

BerekenBrutoTheoretischJaarLoon
  Berekent het theoretisch jaarloon (voor BV-barema bepaling)
  Gebruikt door BedrijfsvoorheffingChain als basis

4.3  FASE RSZ: SOCIALE BIJDRAGEN
──────────────────────────────────────────────────

Drie parallelle RSZ-systemen (één per aangifte-type):

A) DMFA (overheid, regulier)
   BerekenSocialeBijdragenDmfa
   BasisRszVerdelerDmfa
   - verdeelt RSZ-basis over opdrachten

B) DMFAPPL (lokale besturen / politie)
   BerekenSocialeBijdragenDmfappl  ← hoofdklasse
   BasisRszVerdelerDmfappl
   DmfapplBijdragePercentages      ← percentages per bijdrage-type
   GewoonLoonDmfappl               ← gewoon loon berekening
   IsPool2TerDmfappl               ← bepaalt pijler 2 toewijzing
   SocialeBijdragenDmfapplVrijwilligeBrandweer
   BerekenVerminderingHerstructureringDmfappl
   WerkbonusKomtInAanmerkingDmfappl
   PensioenPijler2BonusAncienniteitDmfappl
   PensioenPijler2BonusBijdrageDmfappl
   BerekenWerkbonusDmfapplHelper

C) RSZPPO (provincies)
   BerekenSocialeBijdragenRszppo
   GewoonLoonRszppo
   RszppoBijdragePercentages

Gemeenschappelijk:
   AbstractBerekenSocialeBijdragen
     execute():
       1. berekenSocialeBijdragenHoedanigheid(opdracht, opdrachtContext)
       2. berekenSocialeBijdrageVoertuig(...)
       3. controleerRegulBasisRszNegatief(...)
       → vakbondspremie indien KST_VAKBONDSPREMIE_REKENEN = J
       → vrijwillige brandweer kwartaalgrens-controle indien van toepassing

   BerekenTerugbetalingSocialeBijdragen
     Verwerkt terugbetaling van te veel betaalde RSZ

   BerekenBijzondereBijdrage
     Bijzondere bijdrage sociale zekerheid
     Output: BIJZONDERE_BIJDRAGE_KEY, BASIS_BIJZONDERE_BIJDRAGE_KEY

   PostControleRsz
     Controleert RSZ-resultaten na berekening
     Valideert kwartaalgrens vrijwillige brandweer (DMFAPPL + old)

   Gecachede hulpservices (besparen DB-queries):
     CachedRszConstantService        → RSZ-constanten per datum
     CachedRszConstantOldService     → legacy constanten
     CachedRszPlafondPercentageService → plafond-percentages
     CachedDmfapplBasisBijdrageService → DMFAPPL-basis bijdragen

  4. Vermindering personen ten laste (VerminderingBvPersonenTenLaste)
  5. Vermindering werkbonus (BvTools)
  6. Afronden: naar beneden vóór 2023-01-01, standaard daarna
  7. BvVoluntariaat (optioneel, vrijwillige brandweer-component)

4.6  FASE NETTO: NETTOLOON & INHOUDINGEN
──────────────────────────────────────────────────

BerekenNettoLoon
  Formule per LoonElement:
    netto = brutoMaandloon
          + Σ ADDITIONS   (premies, vergoedingen niet-belastbaar, ...)
          - Σ SUBTRACTIONS (RSZ-bijdragen, BV, bijzondere bijdrage, ...)
    Uitzondering: AFTROK_EVUD_VWG (enkelvoudig vakantiegeld aftrok)
                  → bruto telt NIET mee voor netto
  Output: NETTO_KEY per LoonElement

BerekenInhoudingen
  Verwerkt alle inhoudingen (vakbondsbijdrage, alimentatie, ...)
  Leest InhoudingTypes via LoonElement-eigenschappen

BerekenWerkuitkeringInhoudingen
  Specifiek voor werkloosheidsuitkeringen
  Berekent inhoudingen op vervangingsinkomen

BerekenBeslagOpLoon
  Berekent beslag op loon (juridisch beslag)
  Complexe regels: beschermd deel, gezinssamenstelling
  Input: beslagbedrag via Opdracht.getOpdrachtAanvullendeGegevens()

FixManueleTussenkomstBeslagOpLoon
  Corrigeert het netto-loon indien er een manuele tussenkomst is
  voor beslag op loon (RegulManager-kode check)

4.7  POST-FASE: SOMMATIES & CONTROLES
──────────────────────────────────────────────────

Sommaties
  Per opdracht: sommeert bedragen, belastbaar, inhoudingen
  Per persoon:  sommeert alle opdracht-totalen
  Output: alle OPDRACHT_BEDRAG_KEYS + PERSOON_BEDRAG_KEYS ingevuld

SommatiesAchterstal
  Zelfde als Sommaties maar voor achterstalperioden

PostRekenControles
  Valideert eindresultaten:
  - Negatief nettoloon
  - Onlogische combinaties BV > belastbaar

5. ACHTERSTAL-VERWERKING
================================================================================

Achterstalberekening is een aparte ketting parallel aan de hoofd-ketting:

  RekenAchterstalChain
    └─ AchterstalPreOmschakelingChain
       - Herberekent loon voor vroegere perioden
       - Zet context-vlag ACHTERSTAL = true
       - Voert dezelfde BV-berekeningen uit maar met histori
         sche fiscale constanten (taxdatum van achterstalperiode)

  BelastbaarAchterstalChain
    └─ BerekenBelastbaarAchterstal
       - Berekent belastbaar van de achterstal als éénmalige vergoeding
       - Verwijst naar BerekenBelastbaarVrijwilligeBrandweer indien relevant

  BedrijfsvoorheffingAchterstalChain
    └─ BerekenBvAchterstal
       - Herberekent BV op achterstal met correcte fiscale tabel
       - Alle BV-types hebben een ACHTERSTAL-sleutel variant

  OvernemenBijzondereBijdrageChain
    └─ OvernemenBijzondereBijdrage
       - Kopieert bijzondere bijdrage van achterstalperiode

  ActiveerAchterstal
    - Vlag-command dat de achterstal-status activeert in de ketting
    - Wordt getriggerd als er achterstalperioden zijn

6. APO (AANVULLEND PENSIOEN OVERHEIDSSECTOR)
================================================================================

APO = pensioenopbouw buiten het RSZPPO-systeem
Eigen subcyclus met aparte chain:

  VerwerkApoInput
    - Converteert APO-invoer naar RekenContext
    - ApoGegevensConvertor: mapt APO-data naar Money-sleutels

  ApoEarlyStopCheck
    - Kort-circuiteert de ketting als APO niet van toepassing is

  ApoInputChain
    - Verwerkt APO-bijdragen (werkgever + werknemer)

  ApoNettoEnInhoudingenChain
    - Berekent netto-effect van APO op het loon
    - VerwerkApoInputInhoudingen: APO-specifieke inhoudingen

  Output-sleutel: NIET_BELASTBAAR_APO_KEY (in RekenBbKey)
  Toepassing: overheids-mandatarissen met aanvullend pensioenplan

7. SLEUTELPATRONEN & DESIGN
================================================================================

7.1  TYPED KEY-VALUE STORE (RekenContext)
  - RekenContext = HashMap<ContextKey, Object> met type-safe get/put
  - Keys zijn enums die ContextKey implementeren
  - MoneyKey is een sub-interface voor Money-waarden
  - Hiërarchie: mainContext > opdrachtContext > LoonElement.property

  Key-families:
    RekenBrutoKey    → BRUTO_MAANDLOON_KEY, BASIS_SOCIALE_INHOUDINGEN_KEY
    RekenRszKey      → RSZ-bijdragen, BIJZONDERE_BIJDRAGE_KEY
    RekenBasisRszKey → basis voor RSZ-berekening
    RekenBbKey       → belastbaar inkomen keys
    RekenBvKey       → alle 14+ BV-keys + ACHTERSTAL variants
    RekenNettoKey    → NETTO_KEY
    RekenInfoKey     → informatieve waarden (niet-financieel)
    RekenChainKey    → chain-interne objecten (managers, persoon, ...)
    RekenPatronaalRszKey    → patronale RSZ
    RekenPatronaalPriveRszKey → patronale RSZ privé-sector

7.2  MONEY-TYPE
  - Alle bedragen als Money (wrapper rond BigDecimal)
  - Money.ZERO als default (null-safe)
  - round(2) voor eindresultaten
  - GRENS_INFINITY = Money("99999.99") als bovengrens in barema

7.3  CACHING-PATROON
  - Gecachede services (CachedRszConstantService, ...) besparen DB-queries
  - Cache per RekenTijd (maand + jaar)
  - AbstractCachedRegimeRszppoService als basis

7.4  REGUL-PATROON
  - RegulManager beheert "regulaties" (correcties/aanpassingen)
  - Regulkode enum definieert de types
  - Gebruikt voor: negatieve RSZ-basis, beslag, manuele tussenkomsten

7.5  LOONELEMENT-PATROON
  - Elk loon-type = een LoonElement
  - LoonElement heeft AardKodeExtensie (looncode-groep)
  - Groepering via LooncodeGroepKeyStrings (bijv. AFTROK_EVUD_VWG_KEY)
  - LoonElementManager beheert collecties per opdracht en persoon
